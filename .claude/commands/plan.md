# /plan — 체계적 계획 수립

지금부터 "$ARGUMENTS"에 대한 구현/수정/리팩토링 계획을 세운다.
**핵심 규칙: 이 절차 동안 계획 파일 외의 소스 코드는 수정하지 않는다. 사용자가 명시적으로
"실행"/"진행"을 승인하기 전까지 구현에 착수하지 않는다.**

## 1. 대상 정의
- 무엇을·왜 하는지 한두 문장으로 재진술. 모호하면 먼저 사용자에게 질문한다.
- 완료 기준(무엇이 되면 끝)과 범위 밖(하지 않을 것)을 분명히 한다.

## 2. 상세 분석 (읽기 위주, 추측 금지)
- 관련 코드를 실제로 읽고 `file:line`으로 근거를 남긴다.
- 규약은 `Refactoring/docs/conventions.md`, 이론은 `Refactoring/docs/WIKI.md`,
  아키텍처는 해당 `CLAUDE.md`를 참조한다.
- 레퍼런스(논문 등)가 있으면 핵심 알고리즘·수식·게인 구조를 요약한다.
- 필요하면 `matlab -batch`로 규약·수치를 실제 확인한다.

## 3. 계획 파일 작성
- 편집 가능한 파일로 쓴다: `plans/<slug>.md` (사용자가 직접 열어 수정하도록). 이 경로는 gitignore됨.
- 구성:
  - 배경/목표, 완료 기준, 범위 밖
  - 영향 받는 파일 목록
  - 단계별 작업 (각 단계: 무엇을·어디를·어떻게 + 위험/난이도)
  - 각 단계의 검증 방법 (가능하면 matlab 실행으로 확인)
  - 열린 질문 / 결정 필요 사항

## 4. 사용자 리뷰 루프
- 계획 파일을 제시하고 피드백을 요청한다.
- 사용자가 파일을 직접 수정하거나 코멘트하면 계획을 갱신한다.
- 확정될 때까지 반복한다. **아직 구현하지 않는다.**

## 5. 실행 (승인 후에만)
- 사용자가 명시적으로 승인하면 계획 단계 순서대로 구현한다.
- 각 단계 완료 시 계획 파일의 상태를 갱신한다(체크 표시).
- 비자명한 변경은 완료 후 검증(가능하면 matlab 실행)까지 수행한다.
