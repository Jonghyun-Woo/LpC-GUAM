# /plan — 체계적 계획 수립

지금부터 "$ARGUMENTS"에 대한 구현/수정/리팩토링 계획을 세운다.
**핵심 규칙: 이 절차 동안 계획 파일 외의 소스 코드는 수정하지 않는다. 사용자가 명시적으로
"실행"/"진행"을 승인하기 전까지 구현에 착수하지 않는다.**

## 1. 대상 정의
- 무엇을·왜 하는지 한두 문장으로 재진술. 모호하면 먼저 사용자에게 질문한다.
- 완료 기준(무엇이 되면 끝)과 범위 밖(하지 않을 것)을 분명히 한다.

## 2. 탐색 (읽기 위주, 추측 금지)
- `docs/` 하위 설계 문서를 먼저 읽고 기획·아키텍처·설계 의도를 파악한다:
  - 무엇을·왜: [`docs/PRD.md`](../../docs/PRD.md)
  - 구조/클래스 관계: [`docs/ARCHITECTURE.md`](../../docs/ARCHITECTURE.md)
  - 과거 설계 결정: [`docs/ADR.md`](../../docs/ADR.md)
- 규약은 `Refactoring/docs/conventions.md`, 이론은 `Refactoring/docs/WIKI.md`,
  아키텍처 세부는 해당 폴더의 `CLAUDE.md`를 참조한다.
- 관련 코드를 실제로 읽고 `file:line`으로 근거를 남긴다. 탐색 범위가 넓으면 Explore 에이전트를 병렬 사용한다.
- 레퍼런스(논문 등)가 있으면 핵심 알고리즘·수식·구조를 요약한다.
- 필요하면 `matlab -batch`로 규약·수치를 실제 확인한다.

## 3. 논의
- 구체화하거나 기술적으로 결정해야 할 사항이 있으면 사용자에게 제시하고 논의한다.
- 열린 질문·트레이드오프를 먼저 좁힌 뒤 계획 파일 작성으로 넘어간다.

## 4. 계획 파일 작성
- 편집 가능한 파일로 쓴다: `plans/<slug>.md` (사용자가 직접 열어 수정하도록). 이 경로는 gitignore됨.
- 구성:
  - 배경/목표, 완료 기준, 범위 밖
  - 영향 받는 파일 목록
  - 단계(step)별 작업 — 아래 **Step 설계 원칙**을 따른다.
    각 step은 **작업 내용 + 검증 방법**을 한 쌍으로 반드시 함께 적는다 (검증 없는 step은 미완성이다).
  - 열린 질문 / 결정 필요 사항

### Step 설계 원칙
1. **Scope 최소화** — 하나의 step에서 하나의 레이어/모듈만 다룬다. 여러 모듈을 동시에 손대야 하면 step을 쪼갠다.
2. **자기완결성** — 각 step 기술은 그 자체로 읽고 실행 가능해야 한다. "앞서 논의한 대로" 같은 외부 참조 대신 필요한 경로·맥락을 step 안에 적는다.
3. **사전 준비 강제** — 각 step에 읽어야 할 문서 경로와 이전 step에서 만든/고친 파일 경로를 명시한다.
4. **시그니처 수준 지시** — 함수/클래스의 인터페이스만 제시하고 내부 구현은 재량에 맡긴다. 단, 벗어나면 안 되는 핵심 규칙(수치 규약, 프레임 정합성, 데이터 무결성 등)은 반드시 박아넣는다.
5. **검증은 모든 step의 필수 요소** — 예외 없이 각 step은 자체 검증 방법을 포함한다. "~가 동작해야 한다" 같은 추상 서술이 아니라 실제 실행 가능한 커맨드로 적고, 무엇을 보면 통과인지(기대 출력·수치·거동)까지 명시한다. 이 프로젝트는 빌드/테스트 프레임워크가 없으므로 MATLAB 실행으로 검증한다. 검증 강도는 step의 위험도에 맞춰 아래에서 고른다:
   - 스모크: run-transition-sim 스킬 (또는 `matlab -batch "run('Refactoring/run_transition_sim.m')"`)
   - 부분 확인: `matlab -batch "<검증 스니펫>"` — 기대값/판정 기준 함께 기재
   - **전용 검증 스크립트 작성**: 인라인 스니펫으로 부족한 step(경계조건·수치정합·회귀 비교 등)은 검증용 `.m` 스크립트를 만드는 것을 그 step의 작업 산출물로 포함한다. 계획에 스크립트 경로(예: `Refactoring/tests/verify_<slug>.m`), 무엇을 검증하는지, PASS/FAIL 판정 로직(기대값·허용오차·원본 대비 비교 등)을 명시한다. 스크립트는 명확한 PASS/FAIL을 출력하도록 작성한다(예: `assert` 또는 통과 메시지 + 실패 시 error). 코드이므로 스크립트 본문·주석은 영어로 쓴다.
   - 실행 검증이 불가능한 step(순수 문서·설정 등)이라면 그 사유와 대체 확인 방법(예: 수동 리뷰 체크리스트)을 명시한다.
6. **주의사항은 구체적으로** — "조심해라" 대신 "X를 하지 마라. 이유: Y" 형식으로 적는다.
7. **네이밍** — step slug와 계획 파일 slug는 kebab-case로, 핵심 모듈/작업을 한두 단어로 표현한다 (예: `guam-cbf`, `aero-model`, `control-alloc`).

## 5. 사용자 리뷰 루프
- 계획 파일을 제시하고 피드백을 요청한다.
- 사용자가 파일을 직접 수정하거나 코멘트하면 계획을 갱신한다.
- 확정될 때까지 반복한다. **아직 구현하지 않는다.**

## 6. 실행 (승인 후에만)
- 사용자가 명시적으로 승인하면 계획 단계 순서대로 구현한다.
- 각 단계 완료 시 계획 파일의 상태를 갱신한다(체크 표시).
- 비자명한 변경은 완료 후 검증(가능하면 `matlab -batch` 실행)까지 수행한다.
